Evaluation case study · speech recognition

Code, a local model, or a frontier model?

Choosing a detector for the most common failure in a private, on-device storytelling transcriber — and measuring exactly what the privacy and hardware constraints cost in accuracy.

The short version

What we evaluated
The single most frequent failure in a local-only family-storytelling transcription system: proper names mis-transcribed — above all, the child protagonist's own name.
The method
Build a human ground-truth answer key, then score three classes of detector against it — deterministic code, a range of local language models (2B–26B), and a frontier model — on recall, precision, and reliability.
What it found
A few lines of deterministic code catch ~97% of the errors, with no model at all. Local language models add nothing and are less reliable. A frontier model catches the handful of reasoning-dependent cases — but can't be deployed, because the system is private by design. The local models capable enough to match it don't fit on consumer hardware.
The decision
Ship the code. Name the 1–2 cases it can't reach as the honest, measured cost of the privacy and hardware constraints — not as something to paper over.

1The system, and the failure worth fixing

The system is a small pipeline that records a parent and child telling bedtime stories, transcribes the audio with speaker labels, and pulls out the recurring people and places. One rule shapes everything: it is local-only by design — family audio never leaves the device, and no cloud service ever sees it.

An error-analysis pass over the sessions surfaced a clear top failure mode: names. The transcriber confidently mis-spells the people in the stories — and the most-mangled name of all is the story's own protagonist, the child. It is the kind of error that quietly poisons everything downstream: a "bible" of characters and themes is worthless if the lead character is spelled four different ways.

One property of the system matters for everything that follows. It is a recognition system, not a generation system. Almost every error is "the transcript doesn't match the audio." That single fact decides who can even be a candidate detector — because a text-only judge never hears the sound.

2The eval, in four moves

The framework is deliberately plain, and it is the point. Evals are structured data work on a stochastic system; the rigor lives in the bookkeeping, not in cleverness.

  • A rubric / answer key. A human (the domain owner) assembled the ground truth: each real recurring person → their one correct spelling → every way the transcriber actually mangled it, per session. This is the thing every detector is graded against.
  • A test harness. A small, read-only script that runs any candidate detector across every session and produces flags — re-runnable, deterministic to invoke, and unchanged between candidates so the comparison is fair.
  • A scorecard. Precision and recall for each detector, plus a reliability column (does it crash? does it answer the same way twice?).
  • Reading the off-diagonals. Not the headline percentage — the specific cells. Which true errors did it miss, and which non-errors did it invent? That is where every real decision was made.

3Four shapes of one failure

"Name errors" is not one problem. The answer key revealed four distinct shapes, and they do not yield to the same tool. The distinction that organizes them: does the wrong output leave a trace in the text, or only in the audio?

The four error shapes, and where the evidence lives
ShapeWhat it looks likeWhere the "it's wrong" signal is
Mis-spelling, consistentOne wrong spelling, used the same way every timeIn the text — but only versus the correct spelling (it looks internally fine)
Mis-spelling, inconsistentThe same name spelled several ways in one storyIn the text — the spellings contradict each other
Front-distortionA surname whose opening sound is misheard, so it no longer sounds like the real name at allFaint — sound-matching can't bridge it; needs the known name or reasoning
Dissolved into wordsThe name rendered as an ordinary, correctly-spelled phrase — or a different, perfectly plausible nameOnly in the audio. The text reads as flawless and unremarkable
A detector can only flag an error that leaves a trace in something it can look at. Three of these shapes leave a trace in the text. The fourth erases the very thing detection keys on — it turns a name into a normal phrase — so no text-only method, however clever, can ever catch it. That is not a weakness to engineer around; it is a fact about where the information is.

4The candidates, and the scorecard

Candidate A — plain code

The first candidate is not a model at all: convert each transcript word and each known name to a rough phonetic code, and flag any word that sounds like a name but isn't spelled like one. It is roughly fifteen lines. Scored against the answer key, it lands at ~0.96 precision and ~0.97 recall — and, crucially, it catches the "consistent mis-spelling" case that pure inconsistency-checking misses, because it compares against the correct answer, not against a word's neighbors. It is deterministic: same input, same output, forever. Its only blind spots are the front-distortion case (the sound diverged too far) and the dissolved case (no text signal exists).

Candidate B — local language models

The intuitive hope is that a language model, handed the list of real names, would simply read the transcript and catch everything. We tested that directly, across a ladder of on-device models, scoring each against the same answer key.

Two failure modes recurred. The smaller models crash: handed a long, name-dense session, they try to "correct" every proper noun, loop on the invented ones, and overflow. The mid-size models stop crashing but over-flag — they force ordinary words and made-up character names onto the real-name list with serene confidence. And every model that comfortably runs on the hardware misses the reasoning cases entirely — or notices something is off but assigns it to the wrong person.

Candidate C — a frontier model

A frontier model, tested as a fresh instance with no prior context and the same prompt, did what the local models couldn't. It caught the front-distortion surname (from the line that explained it). It caught the substitute-name case by reasoning — "a parent is thanking the child here, and this name appears nowhere else, so it is probably the child's name mis-heard." It even declined a false positive the local models fell for, reasoning aloud about its own uncertainty. It missed only the dissolved case — the one no text-based method can reach.

5A capability gradient, against a hardware wall

Put end to end, the results are not a binary "LLMs can or can't." They are a gradient, and it runs straight into a wall.

The capability gradient. Format discipline (don't crash) appears around 4B. The easier reasoning case — a distortion the text explains in context — emerges around 12B, and runs locally. The harder reasoning case — an inference the text only implies — needs frontier scale; nothing smaller got it.

The hardware wall. The local models capable enough to close the gap (26B and up) don't fit in a typical laptop's working memory — they load, then run out of memory on real inputs. And there is no convenient middle size to thread the needle.

So the three constraints collide. The capability to catch the reasoning cases requires scale; that scale exceeds the hardware; and the model that has the scale to spare lives in the cloud — which the privacy rule forbids. There is no "capable and local and runs here" option.

6The decision the eval drove

The deployed detector is the code. It is deterministic, it never crashes, it never invents a match, and it catches the overwhelming majority of the errors for the cost of a small script. A local language model — at any size that runs on the device — is strictly worse: it adds no unique catches, it is less reliable, and it hallucinates. The frontier model is better, and undeployable.

What the eval changed is not just the choice of tool; it is the honesty of the residual. The framework lets the system state, with evidence, exactly what its constraints cost:

  • The privacy rule (no cloud) costs roughly one reasoning case a frontier model would otherwise catch.
  • The hardware limit closes the door on the local models that could have caught it on-device.
  • And one case is beyond every model, frontier or local — the name dissolved into ordinary words — because the evidence simply isn't in the text. That one is a job for the audio, or for no one.
The point of an evaluation framework isn't a perfect score. It's the ability to make a constrained decision on purpose, and to say precisely what you traded away to make it. "Ship the cheap deterministic check; here is the 1–2% it can't reach and exactly why" is a complete, honest answer — and a far more trustworthy one than a model that looks impressive and quietly guesses.

A coda: even the cleanup didn't need a model

One question remained. The code's lone false positive was a contraction that simply sounds like the name. Could the smallest on-device model — the one that comfortably fits the hardware — earn its place as a filter, reading each of the code's flags in context and dropping the ones that aren't really names? This is the bounded version of the job — judge a handful of candidates, yes or no — not the open-ended detection that defeated the small models earlier, so it was a fair second chance.

It came back a clean negative. The small model was now perfectly reliable — stable across repeated runs, no crashes; the bounded framing fixed that. But it failed on accuracy, and on exactly the case that was supposed to justify it: a name that is also an everyday word. It correctly dropped the contraction, but it also threw out every instance of a real name that doubles as a common adjective — reading it as the adjective and ignoring the context that made it a name. It traded several real catches for one false-positive removal: a worse detector, not a better one.

What did work was a single line of code: keep only the flags that are capitalized. The stray contraction was lowercase mid-sentence; every real name was capitalized. That one rule removed the false positive and kept every true name — perfect precision and recall, no model, no gigabytes of dependencies. The lesson rhymes with the rest: when a deterministic signal is sitting right there in the data, a model isn't just unnecessary — it can be actively worse, because it brings its own confident mistakes.

7What transfers

The domain here is narrow — recognizing names in family audio — but the method isn't. Three things carry to any evaluation problem:

  • Decompose the failure before you pick a tool. "Name errors" was four different problems wearing one label; the right detector is different for each, and one of them has no software answer at all.
  • A code check often beats an LLM judge. When the error has a mechanical fingerprint, deterministic code is cheaper, faster, perfectly repeatable, and doesn't hallucinate. Reserve the model for the cases that genuinely require reasoning — and prove it earns its place against ground truth before trusting it.
  • Measure the cost of your constraints. Privacy, latency, on-device limits — each one buys you something and costs you something. An eval turns that trade from a vague worry into a number you can defend.
